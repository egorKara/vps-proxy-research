# Proxy (US) — подбор провайдера “точки выхода” трафика: стабильность, цена, тикеты (Obsidian)

---
created: 2026-04-05
tags: [proxy, us, egress, infrastructure]

## 0) Входные параметры (по вашим вводным)

- Гео выхода: **United States**
- Цель: **стабильная инфраструктура** как “конечная точка выхода трафика”
- “Чистота” IP: IP должен быть **приемлемым для AI‑сервисов** (важно: гарантировать это на 100% заранее обычно невозможно — только тестами и политикой замены IP).
- Бюджет: **до \$5/мес** *или* **pay‑as‑you‑go**, если это надёжнее/выгоднее
- Доступ: **HTTP(S) + SOCKS5**
- Предпочтение: **статический IP** (но вы не уверены — ниже есть выбор стратегии под это)

> Примечание по комплаенсу: часть сервисов запрещает доступ через прокси/датацентровые IP или требует определённый регион аккаунта. Рекомендация: использовать прокси только в рамках правил конкретного сервиса.

---

## 1) Ключевая реальность про “чистые IP для AI‑сервисов”

1) Многие AI‑сервисы “не любят” **датацентровые** IP (ASNs крупных хостингов, публичные прокси‑пулы, сильно “зашумлённые” подсети).
2) “Чистота” — это не характеристика провайдера, а **конкретного IP/подсети** *в текущий момент* (репутация меняется).
3) Поэтому главное — не обещания, а:
   - понятная **политика замены IP**,
   - прозрачный тип IP (DC vs ISP/Residential),
   - быстрые **тикеты**,
   - возможность быстро поднять резервную точку выхода.

---

## 2) 2 рабочих подхода

### A) Покупать прокси как сервис (proxy provider)
Плюсы:
- иногда можно выбрать тип IP (ISP/Residential/Datacenter),
- бывает удобная панель, ротации, замены.

Минусы:
- **\$5/мес** почти всегда означает либо **очень мало ресурсов**, либо **shared**‑модель, либо “серую” экономику;
- выше риск получить IP из “уставшего” пула, даже если на лендинге написано “clean”.

### B) Поднять прокси самому на US VPS (self‑hosted egress)
Плюсы:
- стабильнее: **один IP только ваш**, понятная инфраструктура, меньше “шаринга”;
- проще обеспечить безопасность (не стать “open proxy”).

Минусы:
- IP будет **датацентровым** (чаще всего), и “чистота для AI” всё равно не гарантируется;
- нужно базово настроить сервер/прокси и безопасность.

> Для бюджета \$5/мес чаще всего **самый надёжный** путь — B) self‑hosted, и параллельно держать A) как резерв (pay‑as‑you‑go) или второй VPS.

## 3) Критерии выбора (матрица)

### 3.1 IP‑часть (самое важное)
- Тип IP: **Datacenter / ISP / Residential**
- Статичность: IP фиксированный или меняется (и кто управляет)
- Политика замены: **как быстро** и **сколько раз** меняют IP, платно/бесплатно
- “Шаринг”: IP **dedicated** или shared (и сколько соседей)

### 3.2 Управляемость/безопасность
- Есть ли **тикет‑поддержка** и понятный SLA
- Есть ли базовая панель/доступы (если сервис) или console/VNC/rescue (если VPS)
- AUP/abuse: как быстро реагируют на жалобы и что блокируют

### 3.3 Экономика
- стоимость за 1 статический IP / месяц (или минимальный пакет)
- если pay‑as‑you‑go: как считается (за GB/за запрос/за порт)

## 4) Короткий список вариантов под ваш бюджет

### 4.1 Self‑hosted (US VPS) — “1 IP = 1 точка выхода”

**DigitalOcean Droplets (US локации)**
- Базовый план: от **\$4/мес** (512 MiB) и **\$6/мес** (1 GiB)
- На странице pricing в FAQ указано, что у Droplet есть **свой публичный IPv4** (“each Droplet you create comes with its own public IPv4 address”).
- Источник: https://www.digitalocean.com/pricing/droplets

**Linode (Akamai) / аналоги**
- В бюджет около \$5/мес обычно попадает Nanode‑класс (1GB).
- Важно: проверить, включён ли IPv4 и есть ли удобный “rebuild/console”.
- Источник‑агрегатор по цене (проверять по оф. сайту перед покупкой): https://cloudpricecheck.com/linode/linodes-pricing

> Замечание: у крупных облаков IP‑пулы часто “узнаваемые”. Если AI‑сервис чувствителен к датацентровым IP, может понадобиться не VPS, а ISP/Residential‑тип (что обычно дороже).

### 4.2 Прокси‑провайдеры (как сервис) — “быстро взять IP/пул”

**Proxy‑Cheap (US)**
- На странице по US указаны типы, включая Datacenter IPv4 (от **\$1.49/мес**) и Static Residential IPv4 (от **\$2.12/мес**) и поддержка **HTTP/SOCKS5**.
- Источник: https://www.proxy-cheap.com/locations/north-america/united-states

**Webshare (Static Residential / Dedicated Static Residential)**
- У Webshare есть статические ISP/Residential прокси с поддержкой **HTTP/SOCKS5** и разными уровнями “shared vs dedicated”.
- Но экономически “малые бюджеты” там обычно означают **пакеты**, а не “1 IP за \$5”.
Источники:
  - https://www.webshare.io/static-residential-proxy
  - https://www.webshare.io/dedicated-static-residential-proxy

## 5) “Правильные настройки” (рекомендованный baseline)

### 5.1 Если берёте прокси как сервис
- Выбирайте **SOCKS5** там, где нужно проксировать “не только HTTP” (например, разные клиенты/CLI), а **HTTP(S)** — для браузера/простых интеграций.
- Всегда включайте **аутентификацию** (user/pass или IP‑whitelist), иначе повышается риск компрометации.
- Держите **резервную точку выхода** (второй IP/второй провайдер), чтобы не зависеть от “одного IP” при деградации/блокировках.

### 5.2 Если поднимаете self‑hosted прокси на VPS
Безопасность (must‑have):
- **Никогда** не оставлять прокси “открытым в интернет” без пароля/ACL (open proxy).
- Firewall: разрешить вход только с ваших IP (если возможно) или хотя бы поставить строгую auth‑схему.
- Лимиты: ограничить число соединений/скорость (чтобы случайно не стать “ретранслятором”).

Технический baseline:
- Предпочесть один из вариантов:
  - **SOCKS5 server** (например, Dante) — если нужен универсальный транспорт
  - **HTTP(S) proxy** (например, Squid/3proxy) — если в основном браузер/HTTP‑клиенты
- Настроить:
  - таймауты (connect/read) адекватно вашим задачам,
  - keep‑alive (если клиенты делают много запросов),
  - логирование: минимально достаточное для диагностики, без “лишних” данных.

## 6) Как проверять “чистоту” и стабильность без гаданий (24–48 часов)

1) Тест “операционки”: скорость ответа в тикет, понятность ответа, замена IP при проблеме.
2) Тест стабильности: непрерывные короткие запросы (health‑check) + несколько длинных сессий (если ваш кейс это использует).
3) Тест риска “одной точки”: поднять **резерв** (второй IP/второй провайдер) и проверить переключение.
4) Зафиксировать в Obsidian: “провайдер → IP тип → что обещали → что проверили → вывод”.
